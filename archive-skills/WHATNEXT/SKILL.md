# /next-todo — 项目进度审查与下一步规划

## 概述

本流程自动完成 **"读进度 → 审代码 → 选 TODO → 出方案"** 的完整闭环，将分析结果以 `<cwd>/.plan/***_plan.md` 形式输出供用户 Review。

---

## 步骤

### 1. 加载项目上下文

// turbo
读取以下两个核心文件，对齐当前项目状态：

- `PROGRESS.md`：当前迭代目标、已完成项、优先级列表、已知问题
- `Memory.md`：经验教训、踩坑记录、设计原则
- `NEXT_STEP.md`（如存在切未完成）: 当下优先级最高的任务

> **判断**：如果这两个文件不存在或内容过旧，先提醒用户更新后再继续。

### 2. 扫描代码结构

按项目进度需求进行局部的关联项目代码阅读，建立当前实现的全景理解：

1. **顶层目录结构**：`list_dir` 项目根目录和 `src/` 目录
2. **核心模块概览**：对 `PROGRESS.md` 中提到的"进行中"阶段相关的所有源文件执行 `view_file_outline`
3. **测试现状**：`list_dir` `tests/` 目录，了解已有测试覆盖范围
4. **设计文档**：检查 `docs/` 目录下是否有与当前阶段相关的设计文档（如 `PHASE*_DESIGN.md`），如有则阅读
5. **CLI 入口**：检查 `src/*/commands/` 下的命令文件，了解用户侧暴露的功能边界

> **原则**：先 outline 后 detail。只有在需要理解模块间契约（接口/数据模型）时才深入读具体函数。

### 3. 识别理解未完成 TODO

从 `PROGRESS.md` 以及  `NEXT_STEP.md` 的 **"当前优先级"** 和 **"本周执行项"** 中提取所有 `[ ]` 未完成项，并结合代码扫描结果：

1. **验证每个 TODO 的前置依赖**：
   - 它依赖的模块/类/接口是否已实现？
   - 它是否有未完成的前置 TODO？
2. **评估每个 TODO 的粒度**：
   - 能否在一个开发 session（≤2 小时）内完成核心逻辑 + 单元测试？
   - 如果过大，拆分为更小的子任务
3. **排序标准**（优先级从高到低）：
   - P0 > P1 > P2
   - 前置依赖全部就绪的优先
   - 阻塞其他 TODO 最多的优先
   - 已有原型验证的优先

选出 **1 个最小可执行 TODO**。

### 4. 分析依赖与差距

对选出的 TODO 进行深入分析：

1. **输入契约**：它需要哪些模块提供什么数据/接口？逐一确认已实现
2. **输出契约**：它产出什么？谁会消费这些产出？
3. **代码差距**：
   - 哪些是新增文件？
   - 哪些是修改现有文件？具体哪些函数/类需要改动？
4. **已有原型**：是否有 `test_*.py` 或 `examples/` 中的代码可以参考或直接重构？
5. **Memory.md 检查**：是否有与此 TODO 相关的已知坑点或经验教训？

### 5. 设计实施方案

输出标准的 `***_plan.md`，包含：

1. **项目状态表**：当前各模块完成度的表格快照
2. **推荐 TODO 及选择理由**：
   - 为什么是这个？依赖图 / 阻塞关系
   - 预期工作量估算
3. **Proposed Changes**：
   - 按组件分组，标注 `[NEW]` / `[MODIFY]` / `[DELETE]`
   - 新增文件要给出核心类/函数的签名和职责说明
   - 修改文件要说清改动范围
4. **关键设计决策**：
   - 列出 2-3 个设计选择点，说明为什么这样选
   - 标注是否有需要用户确认的决策
5. **Verification Plan**：
   - 单元测试：列出具体测试场景（表格形式）+ 运行命令
   - 手动验证：描述步骤和预期结果

### 6. 录入当前任务PROGRESS.md

将当前的plan reference index录入PROGRESS.md作为参考，引导阅读当前阶段的执行需求

### 7. 输出并等待用户 Review

使用 `notify_user` 将 `***_plan.md` 提交给用户审阅，附上简洁的一句话总结。

> **不要直接开始写代码**。等用户确认方案后再进入 EXECUTION 模式。

---

## 注意事项

- **不修改任何源代码**：本流程是纯分析流程，只产出文档
- **不运行耗时命令**：不执行测试、不启动服务器、不安装依赖
- **尽量详细**：***_plan.md 将作为当前下一步落实技术路线的唯一指南，必须清晰明确
- **尊重 Memory.md**：方案设计必须参考已有的经验教训，避免重复踩坑
